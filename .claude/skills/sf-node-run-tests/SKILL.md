# /sf-node-run-tests — Executable verification

## Role
QA engineer — prove the project is clean, buildable, and runnable with real commands.

## Goal
Run the verification ladder, surface any failure with its output, and confirm a clean state or hand off failures to `/sf-node-fix-issue`.

## Deliverable
A short report in the user's language stating what passed, with command output as proof; or a list of failures.

---

> If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

## Steps

1. **Apply `@rules/verification.md §A`** in order:
   ```
   npm install                  # dependencies resolve
   npm run typecheck            # tsc --noEmit — expect clean
   npm run lint                 # eslint (flat config) — expect clean
   npm run build                # tsup — produces dist/cli.js (shebang, executable)
   npm test                     # vitest run — only if tests were enabled (Phase 1); else skip
   node dist/cli.js --help      # smoke — prints usage, exit 0
   node dist/cli.js --version   # smoke — prints the version, exit 0
   ```
   - **Smoke scope**: `--help` / `--version` exercise the CLI wiring (commander parse, composition root, exit-0 path) **without an org**. The **full** smoke of real `sf` operations (a live `sf data query`, a `sf project deploy`) needs an **authenticated org** reachable through `sf` — that is **out of automatic scope**. State it; never fake it.
   - The static read-through (`@rules/verification.md §B`) confirms every `sf` call goes through `src/sf/runner.ts` via `cross-spawn` (args array, no spawn from a `services`/`commands` module), stdout carries data only, exit codes map at the boundary (`error` → 1 / `warning` → 0 / `ValidationError` → 2), no secret is logged. Tests never spawn the real `sf` — the runner is mocked (`@rules/tests.md`).

2. **Capture output**. A non-zero exit or any reported error is a failure — quote the relevant lines as proof.

3. **Report** (in the user's language, per `@rules/verification.md §C`):
   - All clean → **one short confirmation line** (no full checklist recap), with the proof commands.
   - Any discrepancy → name the command / failing check + the `file:line`, and hand off to `/sf-node-fix-issue` for the root cause (do not patch here unless the user asks).
   - The org-dependent `sf` smoke that could not run → state it plainly and list the exact commands the user must run against an authenticated org.

4. If the **Node/npm environment is not available**: say so plainly, do the static read-through (`@rules/verification.md §B`), and list the exact commands the user must run themselves. **Never claim a clean typecheck / build you could not execute.**

## Test creation policy

- Tests are opt-in at Phase 1. If enabled, the suite (`vitest`) was delivered in its dedicated batch — run it (`npm test`, exit 0).
- **Do not create a test suite if the project has none.** The default verification is typecheck + lint + build + smoke.
- If the user **explicitly asks** to add tests later: follow `@rules/tests.md` (`vitest`, `node` environment, mock `cross-spawn` and the fs read, layer coverage, French test names). Confirm scope, then add `vitest` (dev) + the `"test": "vitest run"` script (+ an optional `vitest.config.ts`).

## Anti-patterns — what NOT to do
- **Do not** report "all green" without having actually run the commands (or without flagging that the env was unavailable).
- **Do not** fix code silently inside this skill — verification reports; fixing is `/sf-node-fix-issue` (root cause) or `/sf-node-add-feature`.
- **Do not** scaffold tests or add a test runner the user did not ask for.
- **Do not** lower a check (skip typecheck, disable a lint rule, drop the smoke) to make the report pass.
- **Do not** claim the org-dependent `sf` smoke passed — it is out of automatic scope; list it for the user.
- **Do not** rewrite the README — this skill is read-only; at most flag it stale for a write skill (`@rules/readme.md`).

## When the user asks something adjacent
- **"Make the build pass"** → if it fails, the root cause is in code/config; classify with `/sf-node-fix-issue`. Do not disable checks to force green.
- **"Add coverage"** → that is explicit test creation; confirm scope + runner, then write tests per `@rules/tests.md`.
