# /sf-node-refactor-code — Restructure without behavior change

## Role
Cautious refactorer — improve structure, preserve behavior, never act without sign-off.

## Goal
Propose a clear restructuring plan, get explicit validation, then apply it with no behavior change.

## Deliverable
A validated plan, then the refactored files on disk + a passing verification + an updated `docs/specs/04-architect.md` if the structure changed.

---

## Hard rule

**No refactor without an explicitly validated plan.** This skill never edits code on first contact. It signals → proposes a plan → waits → only then applies. (Aligned with the global behavior rules.)

> If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

## Steps

1. **Load context**: `docs/specs/04-architect.md` (locked contract), then `@rules/architecture.md` · `@rules/cli.md` · `@rules/errors.md` · `@rules/security.md` · `@rules/sf-cli.md` · `@rules/output.md` (not auto-imported).

2. **Diagnose** what is actually wrong: duplication, a command doing business logic, a god-service, a service that formats output or imports `commander`, a spawn outside `src/sf/runner.ts`, a `Result` → exit-code mapping duplicated instead of at the `cli.ts` boundary, a magic exit number, a `console.log` where an `output/` formatter belongs, a config constant duplicated instead of in `config.ts`, a shared type inlined instead of in `types.ts`, an `sf` command/flag string repeated instead of a typed helper. Anchor each finding to `file:line`.

3. **Decide per finding — factorize OR leave:**
   - **Factorize** when the blocks express the **same intent** (same operation, same data shape), are non-trivial (> ~5 meaningful lines of real logic), and a natural home exists (a typed helper in `src/sf/helpers.ts`, a shared type in `types.ts`, a constant in `config.ts`, a small pure util) **without** forcing a long bag of options or a `mode`/`kind` flag to fork behavior.
   - **Leave** when the blocks are coincidentally similar but conceptually independent, when factoring would cross a layer boundary just to share a few lines, or when the shape will diverge soon. When in doubt, leave it.

4. **Propose the plan** (in the user's language): the findings, the proposed change per finding (factorize/leave + where the helper lives), the files touched, and the explicit promise of **zero behavior change** and **zero security regression** (spawn still only in the runner, args array intact, no secret surfaced, stdout = data / stderr = logs unchanged, exit codes unchanged). Declare any contract deviation the refactor implies (a moved file, a new shared module, a renamed command). Wait for validation.

5. **Apply** only after validation. Minimum diff. Respect the layers, the contract, and `@rules/security.md`.

6. **Verify**: `@rules/verification.md §A` — behavior unchanged, typecheck / lint / build clean, smoke `node dist/cli.js --help` / `--version` unchanged, `npm test` (if tests) still green. If the structure changed (a new shared file, moved code, a renamed command/flag), update `docs/specs/04-architect.md`, regenerate the README (`@rules/readme.md`), and keep the composition in `cli.ts` consistent (every command group registered, every service reachable, the `Result` → exit-code mapping still at the boundary).

## Anti-patterns — what NOT to do
- **Do not** refactor without a validated plan, ever.
- **Do not** factorize two blocks into a helper with a `mode`/`kind` flag that forks inside — that is two functions sharing a body.
- **Do not** create a shared abstraction "for the future" before a second real caller exists.
- **Do not** change behavior under the cover of a refactor — if behavior must change (a new command, a changed flag, a different exit code), that is `/sf-node-add-feature`, declared separately.
- **Do not** weaken security while cleaning up (a spawn moved out of the runner, a value spliced into a command string, a loosened path check, a secret surfaced, a log line pushed to stdout).
- **Do not** move a constant out of `config.ts` or a shared type out of `types.ts` into a local — refactors keep the centralization.
- **Do not** move an `sf` call out of `src/sf/runner.ts` into a service/command, or change a command/flag while refactoring — centralization is the injection guard (`@rules/sf-cli.md`; consult `sf-cli-reference/` by section to keep every command intact).
- **Do not** touch pre-existing dead code unless the user asked to remove it.

## When the user asks something adjacent
- **"Just clean it up however you think is best"** → still propose the plan first; "best" is a judgment the user should confirm before files move.
- **"Refactor and add X"** → split: the refactor (behavior-preserving, validated) and the feature X (`/sf-node-add-feature`) are separate steps with separate validation.
