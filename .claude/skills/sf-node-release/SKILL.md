# /sf-node-release ‚Äî Cut a SemVer release

## Role
Release manager ‚Äî turn the accumulated changes into a clean, versioned, GitHub-ready release.

## Goal
Stamp the pending `[Unreleased]` changelog entries into a dated SemVer version block, bump the version at its single source, and keep every version location consistent.

## Deliverable
Updated `docs/release/CHANGELOG.md` (a new released block + reset `[Unreleased]`), the bumped version source, and a copy-pasteable release note.

---

## Prerequisite

The project must be loaded (`/sf-node-load-project`, OR `docs/specs/04-architect.md` / README present). If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

Load context: `@rules/versioning.md` (+ `@rules/verification.md`).

## Step 1 ‚Äî Read the current state

1. Read `docs/release/CHANGELOG.md`.
   - **Absent** ‚Üí the tool predates the changelog system. Stop and route to `/sf-node-load-project` (it proposes the retroactive initialization). Do not create it here.
   - Present ‚Üí read the `## [Unreleased]` section and the current top released version.
2. Read the canonical version: `package.json` `"version"`.

## Step 2 ‚Äî Nothing to release?

If `[Unreleased]` has no entry: tell the user there is nothing to release (current version = X.Y.Z) and stop. Do not create an empty version block.

## Step 3 ‚Äî Infer and propose the increment

Apply the inference of `@rules/versioning.md`:
1. Any `**BREAKING:**` marker or a `Removed` entry ‚Üí **MAJOR**.
2. Else any `Added` entry ‚Üí **MINOR**.
3. Else (only `Fixed` / internal `Changed`) ‚Üí **PATCH**.

Compute the three candidate versions from the current one. Ask with `AskUserQuestion` (single question), the inferred increment **first and marked `(recommended)`**:

- `MAJOR ‚Üí <x+1>.0.0` ‚Äî [what counts as breaking for a CLI tool, per `@rules/versioning.md`: a renamed/removed command or flag, a changed exit-code semantics, a changed `stdout` data shape, a changed default behavior, a removed config key]
- `MINOR ‚Üí <x>.<y+1>.0` ‚Äî new backward-compatible command(s)/capability
- `PATCH ‚Üí <x>.<y>.<z+1>` ‚Äî fixes / internal changes only

Show the accumulated `[Unreleased]` entries above the question so the user decides with full context. The user has the final call (notably to force a MAJOR the inference cannot detect).

## Step 4 ‚Äî Apply the release

On confirmation, using the chosen version `NEW` and the session date `<YYYY-MM-DD>`:

1. In `docs/release/CHANGELOG.md`: rename `## [Unreleased]` content into a new block `## [NEW] - <YYYY-MM-DD>` (keep the category order Added/Changed/Fixed/Removed/Security), and insert a fresh empty `## [Unreleased]` above it.
2. Bump the **canonical** version: `package.json` `"version"` ‚Üí `NEW`. **That is the only version to write** ‚Äî `src/config.ts` `APP_VERSION` is imported from `package.json` (`import { version } ‚Ä¶`), so it follows automatically; there is nothing else to sync (`@rules/config.md`, `@rules/versioning.md`).
3. Do **not** touch `docs/specs/`, the README, or the tool `CLAUDE.md` origin (that records the framework version, not the tool version).

## Step 5 ‚Äî Verify + hand off

1. Apply `@rules/verification.md` version consistency: the top released version == `package.json` `"version"` (== the derived `APP_VERSION`, no separate mirror to edit); `[Unreleased]` reset empty.
2. Output the cut version block verbatim in a fenced markdown snippet ‚Äî ready to paste into a GitHub release.
3. Remind the user that committing and tagging are theirs to do (this skill never commits): e.g. `git commit -am "release: vNEW"` then `git tag vNEW`.

## Anti-patterns ‚Äî what NOT to do
- **Do not** create the changelog here if it is absent ‚Äî route to `/sf-node-load-project`.
- **Do not** cut a release with an empty `[Unreleased]`.
- **Do not** apply a MAJOR without explicit user confirmation.
- **Do not** hand-edit `src/config.ts` `APP_VERSION` ‚Äî it is derived from `package.json`; bump `package.json` `"version"` only.
- **Do not** rewrite or reword past released blocks ‚Äî only `[Unreleased]` moves.
- **Do not** commit or tag ‚Äî surface the commands, let the user run them.

## When the user asks something adjacent
- **"Just bump to X.Y.Z"** ‚Üí honor an explicit version, but still move `[Unreleased]` into the dated block.
- **"Release and push to GitHub"** ‚Üí do the changelog + version bump; the actual commit/tag/push stays the user's action (surface the commands).
