# /sf-node-trace-feature — Analyze a generated sf-node CLI tool

## Role
Analyst — trace and explain how a command works across the layers, without changing it.

## Goal
Give the user an accurate, layered understanding of a command or behavior, with exact code references — including where the `sf` call happens and where a `Result` maps to an exit code.

## Deliverable
A structured report (in the user's language) with `file:line` references. Optionally `docs/specs/analyse-[topic].md` if the user wants it persisted.

---

> If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

## Steps

1. **Identify scope** — a single file, a command group (command + service + helper), a cross-cutting pattern (`Result` flow, exit-code mapping, stdout/stderr discipline, the config cascade, logging, security, the coupling mode), or the whole tool.

2. **Ask when ambiguous** — if the request has several valid readings, use `AskUserQuestion` before diving in. See "When to ask" below.

3. **Read the relevant sources** by layer:
   - Contract & specs: `docs/specs/04-architect.md` (structure of truth), other `docs/specs/*`.
   - Shared: `src/config.ts` (`AppConfig` + `resolveConfig`), `src/types.ts` (`Result<T>` + DTOs), `src/errors.ts` (named errors), `src/logger.ts` (pino).
   - Composition root / boundary: `src/cli.ts` (commander bootstrap, global handlers, exit-code mapping).
   - Commands: `src/commands/<group>.ts` (thin adapters).
   - Services: `src/services/<domain>.service.ts` (business logic).
   - Salesforce: `src/sf/runner.ts` (the **only** spawner), `src/sf/helpers.ts` (typed helpers), `src/sf/project.ts` (sfdx-project mode only). `sf-cli-reference/INDEX.md` is the command/flag reference, consulted on demand by section — never read whole.
   - Output: `src/output/index.ts` (dispatch) + `csv.ts` / `xlsx.ts` / `table.ts`.

4. **Trace the flow** following the unidirectional path: `cli.ts` (parse) → `commands/<group>` (validate flags) → `services/<domain>` (business rule) → `sf/helpers` → `sf/runner` (`cross-spawn sf [...args, --json]`) → parsed envelope → `Result<T>` → back to the command → `output/` formatter → stdout / file. **Pinpoint the two anchors**: where the `sf` invocation is built (which helper, which catalog section verifies its flags) and where the `Result` / named error maps to an exit code. For a failure path: a service/`sf` layer returns a `Result` failure (`kind: "error"` → exit 1, `kind: "warning"` → exit 0) or raises a named error; the command writes `error.message` to stderr; a thrown error reaches the `cli.ts` boundary and `toExitCode` (`ValidationError` / commander usage → 2, else 1). For `sfdx-project` mode: `cli.ts` resolves the project once at startup via `src/sf/project.ts`.

5. **Security angle** (always include) — verify and report:
   - Process spawn happens **only** in `src/sf/runner.ts` via `cross-spawn`; no `node:child_process` direct, no `shell: true`, no spawn from a `services`/`commands`/`output` module.
   - The `sf` args are an **array** with user values (alias, SOQL, paths) as **discrete elements** — never spliced into a command string.
   - No token/secret is read, stored, or logged; the tool holds at most a non-secret alias; a `Result` `message`/`detail` never carries a secret (`@rules/security.md`).
   - A path from input (`--output`, `EXPORT_DIR`, an import/Apex file) is resolved and confined under the base dir (no traversal); `stdout` carries data only.

6. **Report** (in the user's language):
   - What the code does (behavior).
   - How it connects across layers (dependencies, data flow, the `sf` call, the `Result → exit code` mapping).
   - Any inconsistency vs the contract or the rules (`@rules/*`), with the rule named — including security (`@rules/security.md`).
   - `file:line` references.
   - **Flag any unverified assumption explicitly** — do not fabricate a runtime behavior you could not read (an `sf --json` field shape, a live org response).

## When to ask the user

Use `AskUserQuestion` when:
- **Multiple valid scopes** — "trace the export" could mean the command, the service, the `bulkExport` helper, or the `output/` formatter.
- **Multiple readings** — "how does the query work" — which command? the `sf data query` call? the envelope parse? the `Result` mapping? the formatter?
- **Depth vs breadth** — a deep audit of one command group vs a survey of several.
- **Output format** — short summary, structured report, or actionable findings list.

Offer 2-4 concrete options, recommended one first. Do not ask when the scope is obvious or you are just confirming a fact.

## Anti-patterns — what NOT to do
- **Do not** modify code during an analysis. This skill is read-only; if a fix is wanted, hand off to `/sf-node-fix-issue` or `/sf-node-add-feature`.
- **Do not** report a behavior you inferred but did not read — mark it as an assumption and offer to verify.
- **Do not** describe the layers in the abstract — anchor every claim to a `file:line`.
- **Do not** silently pick one interpretation when several exist — ask.
- **Do not** rewrite the README — this skill is read-only; at most flag it stale for a write skill (`@rules/readme.md`).

## When the user asks something adjacent
- **"And fix it"** → switch to `/sf-node-fix-issue` (bug) or `/sf-node-add-feature` (change), carrying the analysis findings.
- **"Is this secure / well-architected?"** → compare against `@rules/security.md`, `@rules/architecture.md`, `@rules/sf-cli.md`, and `docs/specs/04-architect.md`; report deviations, do not refactor (that is `/sf-node-refactor-code`, on request).
