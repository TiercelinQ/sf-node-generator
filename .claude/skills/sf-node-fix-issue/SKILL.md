# /sf-node-fix-issue — Fix a bug (root cause)

## Role
Debugger — find the root cause and fix it cleanly, never paper over it.

## Goal
Resolve the failure so the relevant `@rules/verification.md` checks pass, with the minimum correct change.

## Deliverable
The corrected file(s) on disk + a passing verification + (after a multi-attempt anomaly) a cleanup report.

---

> If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

## Decision tree — classify the failure first

Read the actual error before touching code (command output, stack trace, the pino log file `logs/app.log`, or the user's repro). Classify, then act in this order:

### 1. Typecheck / lint error (`npm run typecheck`, `npm run lint`)
- Read the exact message + `file:line`. Most are precise (type mismatch, missing import, unused symbol, `any`, extensioned relative import).
- Fix the cause. **Do not** add `// eslint-disable`, `as any`, or `as never` to silence it unless genuinely wrong and justified in a comment. External data (`sf --json`, files, args, env) stays `unknown` then validated — never `any` to make an error vanish.

### 2. Build failure (`npm run build` tsup / `npm run dev` tsx)
- tsup error → an import path, a `.js` extension on a relative import (forbidden under `moduleResolution: "bundler"`), or a TS error surfacing at build — trace to the declaration. The bundle is `dist/cli.js` with a shebang (`banner.js`, `@rules/config.md`).
- The bin is not executable / missing shebang → check `tsup.config.ts` `banner: { js: "#!/usr/bin/env node" }`.

### 3. Runtime exception
- Caught by the global `uncaughtException` / `unhandledRejection` handlers in `cli.ts`? Identify the layer: `sf` spawn / envelope parse → `sf/`; business rule → `services/`; flag parse / dispatch / formatting → `commands/`; file IO (`--output`, exports) → `output/`. Fix at the layer that owns the cause, not where the symptom surfaces. A library layer returns `Result` or raises a named error — it never throws raw at the user.

### 4. `sf` CLI failure (the runner / a helper)
- `ENOENT` on spawn → `sf` not installed or not on PATH; point to the install + `SF_CLI_PATH` / `--sf-cli-path` (config `sfPath`). Confirm the runner uses `cross-spawn` (a bare `node:child_process.spawn` fails on the Windows `sf.cmd` shim).
- Non-JSON / unreadable output → CLI version drift or a call missing `--json` (the runner appends it); verify the flags against the matching `sf-cli-reference/` section (`sf <cmd> --help`).
- `status !== 0` in the envelope → read `message` / `name`, map it to a `Result` error (`kind: "error"`); the command surfaces it on stderr + exit 1; never throw across a layer.
- An invented / misremembered flag → look it up in the catalog section; never guess. See `@rules/sf-cli.md`.
- A Salesforce error in the envelope (an Apex exception, a DML `StatusCode` like `FIELD_CUSTOM_VALIDATION_EXCEPTION`, a `LimitException`, a compile error) → interpret it via `sf-cli-reference/apex-errors.md` to find the root cause (validation rule, missing FLS, SOQL-in-loop, mixed DML…) and craft a clear message (`@rules/errors.md`); keep the raw code in `detail`.

### 5. Security gap (`@rules/security.md` — never "acceptable for now")
- A spawn outside `src/sf/runner.ts`, a concatenated shell string / `shell: true`, a user value spliced into a command string, `node:child_process` used directly → route through the runner with an args array (discrete user values).
- A token/secret written to `.env` / `config.ts` / a log / a `Result` `message`/`detail` → remove it; `sf` keychain only, a non-secret alias at most.
- A path from input (`--output`, `EXPORT_DIR`, an import file, an Apex `--file`) used without the under-base-dir confinement (`resolveWithin`) → add the check (no traversal, no absolute escape). External data used without validating from `unknown` → add the type guard.

### 6. Exit-code / stdout-stderr desync (the CLI I/O contract, `@rules/cli.md`)
- Wrong exit code → the mapping is `warning → 0` / `error → 1` for a returned `Result`, and `toExitCode` (`ValidationError` / commander usage → 2, else 1) for a thrown error, **at the `cli.ts` boundary**. A scattered magic exit number or a `process.exit` outside `cli.ts` is the bug (a command sets `process.exitCode`).
- Data on stderr, or a log / human message on stdout → stdout is **data only** (the `output/` formatter's bytes); stderr carries pino logs + the one-line human message. A `console.log` in delivered code is a defect (only `log.*`).
- A dataset emitted with `console.log(JSON.stringify(...))` → route it through `formatOutput(...)` (`@rules/output.md`).

### 7. sfdx-project mode (only if the coupling mode is `sfdx-project`, `@rules/sfdx-project.md`)
- A missing / invalid `sfdx-project.json` not failing fast → resolve the project once at startup in `cli.ts`, map absence to `ProjectNotFoundError` + a non-zero exit. A hardcoded `force-app` → read `packageDirectories` (honor the `default`). A project-scoped command reaching a `standalone` tool → the mode guard.

## Steps

1. Get the real error (command output, stack trace, `logs/app.log`, or the user's repro). If you cannot reproduce, ask for the exact message — do not guess.
2. Classify with the tree above; read both the failing site and the declaration it depends on.
3. **Ask: "knowing what I know now, what is the clean fix?"** Implement that, not the quickest patch. If the clean fix is larger than expected, say so before applying.
4. Apply the minimum correct change. Respect the layers, the contract (`docs/specs/04-architect.md`), and the security rules.
5. **Verify**: re-run `@rules/verification.md §A` for the affected area (typecheck, lint, build, smoke `--help` / `--version`; `npm test` if tests exist); confirm the targeted failure is gone and nothing else broke. Then apply `@rules/readme.md` — if the fix changed a documented aspect (a command, flag, config key, dependency, tree, coupling mode), regenerate the README. A purely internal fix does not trigger a refresh.
6. If it took several attempts: produce the **cleanup report** (`@rules/architecture.md`), then offer `Do you want to remember this point? /sf-node-save-memory`.

## Cleanup report — multi-attempt anomaly

Once the definitive fix is delivered, list the residue from the failed attempts:

```
Anomaly resolved. Elements to remove from the previous attempts:

File [name]:
- [line / block / import / flag / service call / exit-code branch to delete]
- ...
```

- List only the elements added during the failed attempts that are no longer needed.
- Cover all affected files: TS sources, root configs, `package.json`.
- Deliver the complete cleaned files if the user confirms, then offer `/sf-node-save-memory`.

## Anti-patterns — what NOT to do
- **Do not** silence a typecheck/lint finding with `as any` / `// eslint-disable` instead of fixing the cause.
- **Do not** weaken security to make an error disappear (a shell string, a spawn outside the runner, a logged secret) — fix the actual cause within `@rules/security.md`.
- **Do not** wrap a bug in a `try/catch {}` that hides it — a non-re-throwing catch calls `log.error({ err }, …)`.
- **Do not** route data to stdout mixed with logs, or hardcode an exit code as a magic number — the boundary owns the mapping.
- **Do not** invent an `sf` flag to make a call pass — verify against `sf-cli-reference/`.
- **Do not** leave failed-attempt residue in the file. Clean it up.
- **Do not** expand the fix into an unrequested refactor; flag genuinely-wrong surrounding code for `/sf-node-refactor-code`.
- **Do not** claim it is fixed without re-running the verification (or telling the user which command to run if no env).

## When the user asks something adjacent
- **"Just make the error go away"** → the error is a symptom; explain you are fixing the cause. A genuine temporary silence is explicit and flagged as debt.
- **"It's slow / OOMs on a big export"** → not a crash; prefer the `bulkExport` helper (`sf data export bulk`, server-side pagination) or a streaming formatter (`csv` / `xlsx` to a file) over an in-memory `query` — handle as a finalization or `/sf-node-refactor-code` item, on request (`@rules/output.md`).
